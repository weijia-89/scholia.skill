## scope

Chapter 2 defines **nonfunctional requirements (NFRs)** for data-intensive systems via a Twitter/X-style social network case study (lines 1633–3156). Four pillars:

1. **Performance** — response time vs throughput; latency vs service time vs queueing; percentiles; load parameters; metastable failures and retry storms; backoff, circuit breakers, load shedding.

2. **Reliability and fault tolerance** — faults vs failures; hardware/software/human errors; fault-tolerant design; testing; chaos engineering; security briefly noted out of scope.

3. **Scalability** — load description; vertical vs horizontal scaling; shared-nothing architectures; elasticity; fan-out/on-read vs fan-out/on-write timelines.

4. **Maintainability** — operability, simplicity, evolvability; technical debt; big ball of mud; domain-driven design preview.

Case study threads through performance and scalability sections (500M posts/day, celebrity fan-out). [verified from text, lines 1633–3156]

## key_findings

1. **Functional vs nonfunctional** — Speed, reliability, compliance, maintainability as load-bearing requirements equal to features. [verified, 1656–1667]

2. **Social network scale assumptions** — 500M posts/day (~5.7k/s avg, 150k/s spikes); avg 200 follows; celebrity skew. [verified, 1703–1708]

3. **Relational schema baseline** — users, posts, follows tables as starting design. [verified, 1710+]

4. **Home timeline fan-out** — Materialized timelines per user; write amplification on post; celebrity special-casing. [verified, 1750–1807]

5. **Response time vs throughput** — User-facing latency vs resource/cost sizing; queueing links them (Figure 2-3). [verified, 1809–1878]

6. **Metastable failure** — Overload → timeouts → retries → worse overload; may need reboot. [verified, 1846–1857]

7. **Overload controls** — Exponential backoff, circuit breaker, token bucket, load shedding, backpressure. [verified, 1859–1867]

8. **Latency terminology** — Response time includes network/queueing; service time is active processing; latency catch-all for waiting. [verified, 1880–1899]

9. **Percentiles over averages** — p50/p95/p99/tail; SLO thinking; histograms. [verified, 1900+ range]

10. **Describing load** — RPS, fan-out ratio, read/write mix, payload size as scalability inputs. [verified, 2427+]

11. **Reliability definition** — Continue working correctly despite faults; fault ≠ failure. [verified, 2064+]

12. **Fault categories** — Hardware (disk, RAM, power, network), software bugs, human config errors; correlated failures. [verified, 2100+ range]

13. **Fault tolerance tactics** — Redundancy, rolling upgrades, graceful degradation, monitoring, chaos experiments. [verified, 2200+ range]

14. **Scalability** — Ability to increase throughput by adding resources; limited by bottlenecks. [verified, 2384–2426]

15. **Elastic vs scalable** — Cloud elasticity automatic; scalability architectural. [verified, 2450+ range]

16. **Maintainability triad** — Operability (run in prod), simplicity (manage complexity), evolvability (change ease). [verified, 2563+]

17. **Evolvability** — Rolling upgrades, schema evolution foreshadow Ch. 5; DDD/agility references. [verified, 2700+ range]
