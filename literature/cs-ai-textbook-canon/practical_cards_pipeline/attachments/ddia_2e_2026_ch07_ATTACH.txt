## scope

Chapter 7 covers **sharding (partitioning)** when data exceeds one machine (lines 11679–12807). Combines conceptually with Ch. 6 replication but treats sharding independently for clarity. Major arcs:

1. **Terminology** — Shard vs partition vs region/tablet/vnode; PostgreSQL partition-on-one-machine vs cross-machine sharding.

2. **Goals** — Even load spread; avoid skew/hot shards/hot keys; rebalance on cluster membership change.

3. **Key-range sharding** — Sorted ranges; efficient range scans; hot spots on adjacent keys (timestamp anti-pattern); split/merge rebalance.

4. **Hash sharding** — Hash partition; consistent hashing; fixed shard count; virtual nodes; destroys range query locality.

5. **Composite keys** — Partition key prefix + sort suffix for range-within-shard.

6. **Multitenancy** — Shard-per-tenant vs pooled tenants.

7. **Secondary indexes** — Local (scatter-gather reads) vs global (multi-shard writes); ties to Ch. 4 inverted indexes/Elasticsearch routing.

8. **Request routing** — Coordinator; ZooKeeper/etcd cluster membership.

Multi-shard transactions deferred to unavailable later chapters. [verified, lines 11679–12807]

## key_findings

1. **Two distribution modes** — Replication (same data, many nodes) vs sharding (split data). [verified, 11701–11710]

2. **One shard per record** — Record belongs to exactly one shard; replication still copies shard. [verified, 11712–11722]

3. **Replication + sharding combo** — Per-shard leaders/followers (Figure 7-1). [verified, 11724–11733]

4. **Naming proliferation** — Kafka partition, Cockroach range, Cassandra vnode, etc. [verified, 11740–11754]

5. **Even distribution goal** — 10 nodes → ~10× capacity; rebalance on add/remove. [verified, 11915–11924]

6. **Skew / hot shard / hot key** — Celebrity problem; single busy node bottleneck. [verified, 11926–11932]

7. **Key-range sharding** — Encyclopedia volume analogy; non-uniform ranges; manual (Vitess) or automatic (Bigtable, HBase, MongoDB, CockroachDB). [verified, 11941–11969]

8. **Range scan advantage** — B-tree/SSTable within shard; sensor timestamp example. [verified, 11971–11979]

9. **Timestamp hot spot** — Monthly shard gets all writes; fix with sensor-ID prefix sacrificing cross-sensor range query. [verified, 11981–11996]

10. **Range rebalance** — Pre-splitting; split on size/throughput; merge on delete. [verified, 11998–12015]

11. **Hash sharding** — Even spread; loses ordering; consistent hashing (Karger et al.); Java hashCode not safe distributed [16]. [verified, 12039+]

12. **Fixed shards on nodes** — Move whole shards; virtual nodes; Cassandra token algorithms. [verified, 12100+ range]

13. **Warehouse partitioning** — Different from OLTP; parallel aggregate across shards [Link to Come]. [verified, 12185+]

14. **Local secondary indexes** — Index only within shard; write one shard; read all shards (scatter-gather). [verified, 12437–12479]

15. **Global secondary indexes** — Separate sharding of index; multi-shard writes; single-shard postings read. [verified, 12480+]

16. **DIY secondary index warning** — Race conditions cause drift; ties to Ch. 4 postings lists. [verified, 12481–12488]

17. **Routing** — Coordinator tracks shard→node map. [verified, summary 12642–12644]

18. **Cross-shard writes** — Partial failure problem; forward to Ch. 8+ **[unavailable]**. [verified, 12646–12650]
