## Scope

Chapter 9 positions **LLM workflows** on a **generality vs strength** spectrum (Figure 9-1): conversational agents are general but weak on multi-step unattended work; workflows trade generality for structured, high-fidelity task execution. AGI contrast frames current LLM limits (reasoning, math, no post-training learning). Framework discussion deliberately avoided (LangChain, AutoGen, etc.)—patterns are portable.

**Would a Conversational Agent Suffice?** uses the **Shopify plug-in promoter** running example: scrape storefronts → extract details → ideate plug-in → email owners → send. Pure chat fails; tool-equipped chat still naive (web search "best Shopify 2024", `[your_name]` templates); narrowing system message + custom tools increases prompt bloat and lacks queue/unit-of-work semantics—motivating workflows.

**Basic LLM Workflows** five-step meta-process (define goal → specify tasks → implement → connect → optimize). **Tasks**: explicit I/O schemas (Tables 9-1/9-2 plug-in and email fields); templated prompt approach (Table 9-3 prefix/suffix email template with post-processing); **tool-based extraction** (Table 9-4 restaurant HTML → structured tool call; `tool_choice` force-call; structured outputs). Failure modes: human-impossible extraction, overly nested schemas—decompose. **Sophistication**: CoT/ReAct in tasks; disable function calling first turn (`tool_choice: "none"`) for planning; **Reflexion** subtask on failed output; expert+UserProxy agent pattern (AutoGen mention). **Add variety**: crawlers/DB/BERT where sufficient; human approval for irreversible actions; per-task model sizing.

**Assembling the Workflow**: state machine / pub-sub / orchestrator views; **pipeline** (Figure 9-5) vs **DAG** (Figure 9-6, storefront details to both ideation and email) vs **cyclic graph** (QC loop, Figure 9-7)—complexity warnings (reunite failure context, attempt limits, hide recursion in tasks). Batch vs streaming processing.

**Example Workflow: Shopify Plug-in Marketing**: mock HTML emit → LLM summarize (6 questions) → two-step brainstorm+concept → multistep email (strategy, subject, body) → mock send; Fly By Jing sample output; optimization notes (idea variety, feasibility subprocess, Reflexion, I/O harnesses, DSPy/TextGrad, production A/B—Ch. 10 forward ref).

**Advanced LLM Workflows**: LLM-driven routing (workflow as agent with task tools; agent-of-agents; dynamic task spawning; work-list prioritization); **stateful task agents** (file-bound agents, dependency notifications, user↔agent file discussion); **Roles and Delegation** (AutoGen Assistant/UserProxy, group chat manager; CrewAI sequential/hierarchical/consensual).

## Key findings

All claims below are **[verified from text]** unless tagged `[contested in chapter]` or `[inferred]`.

### Generality–strength tradeoff

- LLMs excel at breadth but not AGI-level reasoning/math/novel knowledge creation. [verified from text, lines 8312–8321]
- Conversational agents best for 1–2 step tasks with user assistance; workflows for rigid multi-step goals. [verified from text, lines 8323–8345]
- Framework-agnostic pattern focus intentional. [verified from text, lines 8351–8356]

### Agent failure → workflow need

- Even tool-rich agents produce naive search, form-letter emails, and lack batch queue semantics. [verified from text, lines 8435–8463]
- Each workflow step should be isolated specialized task. [verified from text, lines 8465–8470]

### Task design

- Tasks need defined schemas, implementation strategy, and isolation testability. [verified from text, lines 8544–8592]
- Prefer non-LLM implementation when possible—cost, speed, determinism. [verified from text, lines 8599–8603]
- Templated prompts use prefix/suffix to bound completion for downstream parsing. [verified from text, lines 8663–8669]
- Tool-choice forcing + structured outputs for extraction; split complex schemas. [verified from text, lines 8723–8750]

### Workflow topology

- Pipeline simple but couples data paths; DAG shares upstream outputs to multiple consumers. [verified from text, lines 8889–8924]
- Cyclic graphs for QC possible but increase state/retry complexity—prefer in-task recursion. [verified from text, lines 8930–8962]
- Evaluate tasks in isolation before full workflow integration. [verified from text, lines 8855–8866]

### Advanced patterns

- LLM can route among fixed tasks, spawn sub-agents, or maintain dynamic work lists. [verified from text, lines 9167–9208]
- Stateful task agents hold persistent work-item state with dependency notifications. [verified from text, lines 9210–9255]
- AutoGen/CrewAI: role-based delegation teams. [verified from text, lines 9257–9302]
- "Simpler is almost always better"; deterministic graph + LLM tasks preferred over LLM-orchestrated chaos. [verified from text, lines 9344–9354]

## Pedagogy

### Learning objectives

1. Place agents vs workflows on the generality–strength spectrum for a use case.
2. Decompose a multi-step business process into tasks with explicit I/O schemas.
3. Implement LLM tasks via templates, tool-based extraction, and sophistication layers (CoT, Reflexion).
4. Choose pipeline vs DAG vs cyclic topology with awareness of coupling and retry costs.
5. Describe advanced patterns: LLM routing, stateful task agents, role delegation.
6. Apply "non-LLM first" and per-task model selection heuristics.
